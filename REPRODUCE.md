# Reproduction guide

Step-by-step replication of the data export underlying the paper.

## Prerequisites

- Node.js 20 or newer (zero npm dependencies; no `npm install` step).
- Bash for `verify.sh` (Linux / macOS / Git Bash / WSL on Windows).
- ~2 GB free disk for the regenerated `bcr_paths_daily.csv` files.

## Step 1: hash-verify the bundled artifacts

```
./verify.sh
```

Expected output ends with `ALL BUNDLED FILES PASS`. Any mismatch indicates a corrupted clone: re-clone or compare the failing file's SHA-256 against the value printed in `verify.sh`.

The script also prints expected hashes for the two `bcr_paths_daily.csv` files that are not bundled. Those checks are skipped if the files don't exist locally; they are enforced if you've run the export step below.

## Step 2: canonical four-step chain (monthly main results)

```
node scripts/preprocess-bitstamp.js
node scripts/export.js
node scripts/build-surface.js
node scripts/build-surface-grid-validation.js
```

This is the paper's canonical chain. Each step depends on the prior step's outputs.

**Step 2a — preprocess-bitstamp.js.** LOCF-imputes the raw Bitstamp daily-close series so that every calendar day carries a price (matters for monthly payment scheduling on non-trading-day month-ends). Writes:

- `data/btc-prices-bitstamp.csv`
- `data/btc-prices-bitstamp-imputation.json`

**Step 2b — export.js.** Sweeps 245 starting BCR values per regime across the four main regimes plus the 2011 supplementary. Reads `data/btc-prices-bitstamp.csv` and `data/btc-prices-coinmetrics.csv`. Writes:

- `outputs/manifest.json`
- `outputs/regime_summary.csv` (1,470 rows)
- `outputs/regime_descriptors.csv` (6 rows)
- `outputs/bcr_paths_monthly.csv` (~48 K rows)
- `outputs/bcr_paths_daily.csv` (~1.3 M rows, ~750 MB)
- `outputs/transition_paths_monthly.csv` (~12 K rows)

**Step 2c — build-surface.js.** Computes per-regime depth-normalized (τ, σ) (population std, ddof=0), pooled (τ, σ) over the three pool regimes (Mt. Gox 2014, ICO Unwind 2018, Fed Tightening 2022), fits the symmetric two-parameter (p, q) family at every (depth, duration) cell, validates against the four observed regimes (gate: max |Δ BCR| ≤ 0.5x), and writes the surface. Merges its outputs into the existing `manifest.json` (no overwrite of `code_version`). Writes:

- `outputs/regime_tau_sigma.csv`
- `outputs/time_at_depth_distributions.csv`
- `outputs/surface_validation.csv` (per-regime peak-date calendar start; retained as a calendar-sensitivity check)
- `outputs/surface_grid.csv` (961 cells, grid calendar start 2017-12-16)

**Step 2d — build-surface-grid-validation.js.** Recomputes the four-regime synthetic required starting BCRs under the grid calendar start (2017-12-16, ICO Unwind 2018 peak, in-pool median across the three pool regimes) so payment-day cadence is deterministic and matches the surface_grid cells displayed in Figures 6 and 7. This is the §6.6.2 body comparison; the prior step's `surface_validation.csv` (per-regime peak-date calendar start) is retained as the calendar-sensitivity companion. Pooled (τ, σ) and the (depth, duration) coordinates are read from the prior step's manifest and CSVs; the surface itself is not rebuilt. Updates `manifest.json` with `phase_2.sha256.surface_grid_validation`. Writes:

- `outputs/surface_grid_validation.csv` (4-row §6.6.2 body table)

## Step 3: daily cadence sensitivity (paper §5.1 footnote 1)

The daily run only needs the first two steps; there is no daily surface.

```
node scripts/preprocess-bitstamp.js
node scripts/export.js --cadence=daily
```

Same outputs as Step 2b, written to `outputs/daily/`. Per-payment obligation = annual / 365 instead of annual / 12. Do not run `build-surface.js --cadence=daily` or `build-surface-grid-validation.js --cadence=daily`: the flag has no effect on either script; the coverage surface and its body-table validation are constructed only at monthly cadence per paper §6.6.

## Step 4: figure-data scripts (paper figures + appendix)

`min_bcr_chart_data_mech.csv` (driving Figures 2 + B1) is generated automatically as the final step of `export.js`; no separate invocation needed. `scripts/derive-mech-chart-data.js` remains standalone-runnable (`--include=covid` adds covid_2020 for ad-hoc plots) but is otherwise wired into the canonical chain.

The canonical paper figures (`figures/fig01`–`fig07`, `figB1`, `figB2`) are mirrored into the repo from the clean-room visualization workspace and are not regenerated by these scripts. See `scripts/README.md` for the script catalog.

## Step 5 (optional): re-derive the preprocessed Bitstamp series

The preprocessed `data/btc-prices-bitstamp.csv` is bundled and produced as Step 2a. To re-derive it from the raw `data/raw/bitstamp_btc_prices.csv` standalone:

```
node scripts/preprocess-bitstamp.js
```

This applies last-observation-carried-forward imputation to fill calendar-day gaps in the raw Bitstamp daily-close series so every calendar month-end carries a price for monthly dividend execution. The imputation audit log is rewritten at `data/btc-prices-bitstamp-imputation.json` (33 imputed days out of 5,344, 2 of them month-ends — both 2011).

The CoinMetrics file (`data/btc-prices-coinmetrics.csv`) requires no preprocessing; it is already calendar-complete daily.

## Step 6 (optional): re-verify after running the export

`verify.sh` will now hash-check the regenerated `bcr_paths_daily.csv` files against the documented expected values and report pass/fail.

## What to do if a hash doesn't match

1. **Bundled file mismatch:** clone integrity issue. Re-clone or download the specific file from the GitHub release.
2. **Regenerated file mismatch after `node scripts/export.js`:** check Node version (engine relies on consistent floating-point semantics; v20+ is verified). If your Node is v20+ and the engine SHA-256 still matches `e29e1d1e…`, file an issue with your `node --version` and the failing file.
3. **Engine SHA mismatch:** you have a modified `src/solvency-engine.js`. Restore from the bundled copy.
